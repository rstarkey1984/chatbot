챗봇/RAG 서비스

LLM 기반 지식베이스(문서셋)별 세션 분리

범위: Spring Boot 백엔드, MySQL, OpenClaw Gateway 연동

목적: 같은 자료로 이어지는 대화는 자연스럽게 연결하고, 다른 자료의 대화는 섞이지 않도록 분리

작성자: 류근철

2. 문제 정의 및 설계 목표

문제 정의

  • 여러 질문이 하나의 흐름으로 이어지며 맥락이 과하게 누적될 수 있음
  • 같은 주제가 끊기면 맥락 단절로 반복 질문이 증가
  • 사용자는 "왜 지금 이 답을 하지?" 같은 혼선을 경험

설계 목표

  • 문서셋 기준으로 세션 라우팅
  • 같은 자료는 이어지고, 다른 자료는 분리
  • 세션 식별값 무결성 보장 + 운영 유연성 확보

3. 개발환경 및 프로젝트 구조

개발환경

  • OS: Ubuntu
  • Java 21, Spring Boot
  • DB: MySQL
  • Node.js
  • OpenClaw Gateway

프로젝트 구조(상세)

  • 프론트엔드: templates(Thymeleaf), static(html/css/js)
  • 백엔드: controller, service, repository, entity
  • 공통 처리: common, controller/advice
  • 데이터베이스: MySQL, sql/ 초기 스키마/운영 SQL
  • OpenClaw Gateway: /v1/chat/completions, agent-id/session-key 기반 라우팅

4. 프로젝트 수행 절차 및 방법

  • 요구사항 분석: 사용자 세션 이력 직접 관리 부담, 맥락 단절/섞임 이슈 확인
  • 시스템 설계: OpenClaw Gateway 도입, user+custom 기준 정의
  • UI/UX 설계: 자료 기반 대화 흐름과 전환 동선 정리
  • 기능 구현: SessionRoutingService, GatewayClient, 요청 검증 반영
  • 테스트/검증: 동일 문서셋 유지, 문서셋 전환 분리, fallback 점검
  • 결과 정리: 라우팅 기준, 로그 포인트, 운영 규칙 문서화

5. 아키텍처/세션 정책/Gateway 요청 계약

  • 아키텍처: ChatQueryService → SessionRoutingService → GatewayClient
  • 세션 모델: sessionId = user + custom (문서셋/주제 기준)
  • 토큰 정책: sessionTicket 기반 무결성 보장, 미제공 시 기본 세션 적용
  • 요청 계약: Authorization, x-openclaw-agent-id, x-openclaw-session-key

6. 시연 단계 (브라우저 기준)

  • 실행: cd ~/projects/chatbot && ./gradlew bootRun
  • 헬스체크: http://localhost:8888/health
  • 채팅: http://localhost:8888/chat?sessionTicket=<A> (문맥 유지)
  • 분리: 새 탭 .../chat?sessionTicket=<B> (세션 분리)
  • fallback: http://localhost:8888/chat
  • 검증: /chat/whoami, /chat/history

7. 기대효과 및 향후 개선

기대효과

  • 같은 문서셋은 이어지고, 다른 문서셋은 분리되어 문맥 혼선 감소
  • 세션 라우팅 기준 명확화로 운영/장애 분석 효율 향상
  • 세션 이력 관리 부담 완화로 유지보수성 확보

향후 개선

  • 세션 라우팅 모니터링 강화(지표, 오류 추적 대시보드)
  • 문서셋 특성 기반 응답 정책 고도화
